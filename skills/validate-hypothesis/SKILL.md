# /validate-hypothesis

비즈니스 모델 가설을 구조화하고, 검증 가능한 문장으로 만든다.

## 트리거

사용자가 다음과 같이 말할 때:
- "가설 세워줘", "가설 기획", "비즈니스 모델 정리", "아이디어 검증 준비"
- "validate hypothesis", "lean canvas"
- `/validate-hypothesis`

## 워크플로우

### Step 1: 아이디어 수집

사용자에게 다음을 질문한다. 한 번에 하나씩, 대화형으로 진행한다.

```
Q1. 무엇을 만들려고 하나요? (한 문장)
Q2. 누가 쓸 건가요? (구체적 타겟)
Q3. 이 사람들이 지금 겪고 있는 문제는 뭔가요?
Q4. 지금은 이 문제를 어떻게 해결하고 있나요? (기존 대안)
Q5. 당신의 솔루션은 기존 대안보다 뭐가 다른가요?
Q6. 얼마에 팔 생각인가요? (가격대)
Q7. 어떻게 돈을 벌 건가요? (수익 모델: 구독, 일회성, 중개 등)
```

사용자가 모르겠다고 하면 → 예시를 들어 힌트를 주되, 대신 정하지 않는다.

### Step 2: Lean Canvas 구성

수집된 답변으로 Lean Canvas 3칸을 채운다:

| 칸 | 내용 |
|----|------|
| **Problem** | Q3 답변 기반. 핵심 문제 1~3개 불릿 |
| **Customer Segment** | Q2 답변 기반. 구체적 페르소나 |
| **Unique Value Proposition** | Q5 답변 기반. "유일하게 [타겟]의 [문제]를 [방식]으로 해결" 형식 |

추가로 다음도 채운다:
| 칸 | 내용 |
|----|------|
| **Existing Alternatives** | Q4 답변 기반 |
| **Revenue Stream** | Q7 답변 기반 |
| **Price Point** | Q6 답변 기반 |

### Step 3: 핵심 가설 도출

Lean Canvas에서 **"이게 거짓이면 사업이 안 된다"는 가설**을 뽑는다.

**XYZ 형식** (Alberto Savoia의 Pretotyping 공식):

```
"최소 [X]%의 [Y]가 [Z]할 것이다."
```

예시:
- "최소 10%의 랜딩페이지 방문자가 가격표를 클릭할 것이다."
- "최소 5%의 랜딩페이지 방문자가 사전등록할 것이다."

가설은 **최대 3개**까지. 그 중 **가장 위험한 1개(Riskiest Assumption)**를 선정한다.

### Step 4: 검증 기준 설정

선정된 가설에 대해 Go/No-Go 기준을 사전 확정한다:

```
📊 검증 기준
- GO:    전환율 [X]% 이상
- PIVOT: 전환율 [Y]~[X]% (메시지/타겟 변경 후 재실험)
- KILL:  전환율 [Y]% 미만
```

기본값 제안:
- GO: 10% 이상
- PIVOT: 3~10%
- KILL: 3% 미만

사용자가 업종/상황에 맞게 조정할 수 있다.

### Step 5: 문서 생성

아래 형식으로 파일을 생성한다.

**파일 경로**: `docs/hypotheses/[slug].md`
- slug는 아이디어명을 kebab-case로 변환

**템플릿**:

```markdown
# [아이디어명] — 가설 문서

> 생성일: [YYYY-MM-DD]
> 상태: 🔬 검증 전

## 한 줄 요약
[Q1 답변 기반]

## Lean Canvas

### Problem
[불릿 리스트]

### Customer Segment
[페르소나 설명]

### Unique Value Proposition
[UVP 문장]

### Existing Alternatives
[기존 대안들]

### Revenue Stream
[수익 모델]

### Price Point
[가격대]

---

## 핵심 가설

### 🎯 Riskiest Assumption
> "[XYZ 가설 문장]"

### 보조 가설
1. [가설 2]
2. [가설 3]

---

## 검증 기준

| 판정 | 조건 |
|------|------|
| ✅ GO | 전환율 [X]% 이상 |
| 🔄 PIVOT | 전환율 [Y]~[X]% |
| ❌ KILL | 전환율 [Y]% 미만 |

---

## 다음 단계
→ `/validate-landing` 으로 랜딩페이지 생성
```

### Step 6: 사용자 확인

생성된 문서를 보여주고 수정할 부분이 있는지 확인한다.
수정 요청이 있으면 반영 후 저장.

## 주의사항

- 사용자 대신 가설을 정하지 않는다. 질문하고 정리하는 역할이다.
- "이게 좋은 아이디어인지" 판단하지 않는다. 그건 데이터가 할 일이다.
- 답변이 모호하면 구체화 질문을 한다. 넘어가지 않는다.
- 가설 문장은 반드시 **측정 가능한 숫자**를 포함해야 한다.
