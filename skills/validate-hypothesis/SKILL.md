# /validate-hypothesis

비즈니스 모델 가설을 구조화하고, 검증 가능한 문장으로 만든다.

## 트리거

사용자가 다음과 같이 말할 때:
- "가설 세워줘", "가설 기획", "비즈니스 모델 정리", "아이디어 검증 준비"
- "validate hypothesis", "lean canvas"
- `/validate-hypothesis`

## 워크플로우

### Step 1: 아이디어 수집

사용자에게 다음을 질문한다. 한 번에 하나씩, 대화형으로 진행한다.

```
Q1. 무엇을 만들려고 하나요? (한 문장)
Q2. 누가 쓸 건가요? (구체적 타겟)
Q3. 이 사람들이 지금 겪고 있는 문제는 뭔가요?
Q4. 지금은 이 문제를 어떻게 해결하고 있나요? (기존 대안)
Q5. 당신의 솔루션은 기존 대안보다 뭐가 다른가요?
Q6. 얼마에 팔 생각인가요? (가격대)
Q7. 어떻게 돈을 벌 건가요? (수익 모델: 구독, 일회성, 중개 등)
```

**대화 중 이미 언급된 정보가 있으면 다시 묻지 않는다.** 파악된 내용을 정리해서 보여주고, 빈 칸만 질문한다.

#### 리서치 모드

사용자가 질문에 "모르겠다", "리서치해줘", "찾아봐줘" 라고 답하면 → **힌트가 아니라 실제 웹 리서치를 수행**한다.

| 질문 | 리서치 수행 내용 |
|------|----------------|
| Q3 (문제) | 타겟 고객의 페인포인트 리서치. 업계 보고서, 커뮤니티, 블로그 등에서 실제 불만/어려움 수집 |
| Q4 (기존 대안) | 현재 시장에서 사용하는 해결 방법 + 대안 서비스/도구 조사 |
| Q5 (차별점) | 사용자의 자산/상황을 분석하여 차별화 포인트 도출. 필요 시 **자산 매핑**(Step 1-b) 수행 |
| Q6 (가격대) | 유사 서비스 가격 벤치마크 리서치 후 구간 제안 |
| Q7 (수익 모델) | 유사 서비스 수익 모델 비교 분석 후 적합한 모델 제안 |

리서치 결과는 정리해서 보여주되, **최종 결정은 사용자가 확인**한다.

### Step 1-b: 자산 매핑 (선택적)

사용자의 조직이 이미 보유한 데이터/인프라/고객 기반이 있는 경우, 이를 비즈니스 모델 차별점으로 연결한다.

```
질문: "현재 보유한 데이터, 고객, 인프라 중 활용할 수 있는 것이 있나요?"
```

있다면 → 보유 자산이 타겟 고객의 어떤 문제를 해결하는지 매핑 테이블을 만든다.

예시:
| 보유 자산 | 이게 알려주는 것 | 타겟 고객에게 줄 수 있는 가치 |
|----------|----------------|---------------------------|
| HR 담당자 연락처 | 의사결정자 접근 경로 | 콜드 아웃리치 없이 직접 연락 가능 |
| 채용 공고 데이터 | 기업의 현재 니즈 | "지금 뭐가 필요한 회사"를 찾아줌 |

이 매핑은 Q5(차별점)와 UVP에 직접 반영한다.

### Step 1-c: 경쟁사 분석 (선택적)

사용자가 경쟁 환경을 모르거나 "경쟁사 있어?", "유사 서비스 있나?" 라고 물으면 수행한다.

리서치 범위:
- **글로벌 유사 서비스** — 같은 문제를 푸는 해외 플랫폼
- **한국 유사 서비스** — 부분적으로라도 겹치는 국내 서비스
- **간접 경쟁** — 다른 방식으로 같은 문제를 해결하는 서비스

결과를 테이블로 정리:
| 서비스 | 하는 일 | 가격 | 한계 (우리와 차이) |
|--------|---------|------|-------------------|

경쟁 갭(우리만 할 수 있는 것)을 명확히 도출하여 UVP에 반영한다.

### Step 2: Lean Canvas 구성

수집된 답변으로 Lean Canvas 6칸을 채운다:

| 칸 | 내용 |
|----|------|
| **Problem** | Q3 답변 기반. 핵심 문제 1~3개 불릿 |
| **Customer Segment** | Q2 답변 기반. 구체적 페르소나 |
| **Unique Value Proposition** | Q5 + 자산 매핑 + 경쟁 갭 기반. 차별점이 명확해야 함 |
| **Existing Alternatives** | Q4 + 경쟁사 분석 기반 |
| **Revenue Stream** | Q7 답변 기반 |
| **Price Point** | Q6 답변 기반 |

### Step 3: 핵심 가설 도출

Lean Canvas에서 **"이게 거짓이면 사업이 안 된다"는 가설**을 뽑는다.

**XYZ 형식** (Alberto Savoia의 Pretotyping 공식):

```
"최소 [X]%의 [Y]가 [Z]할 것이다."
```

예시:
- "최소 10%의 랜딩페이지 방문자가 가격표를 클릭할 것이다."
- "최소 5%의 랜딩페이지 방문자가 사전등록할 것이다."

가설은 **최대 3개**까지. 그 중 **가장 위험한 1개(Riskiest Assumption)**를 선정한다.

### Step 4: 검증 기준 설정

선정된 가설에 대해 Go/No-Go 기준을 사전 확정한다:

```
📊 검증 기준
- GO:    전환율 [X]% 이상
- PIVOT: 전환율 [Y]~[X]% (메시지/타겟 변경 후 재실험)
- KILL:  전환율 [Y]% 미만
```

기본값 제안:
- GO: 10% 이상
- PIVOT: 3~10%
- KILL: 3% 미만

사용자가 업종/상황에 맞게 조정할 수 있다.

### Step 5: 문서 생성

아래 형식으로 파일을 생성한다.

**파일 경로**: `docs/hypotheses/[slug].md`
- slug는 아이디어명을 kebab-case로 변환

**템플릿**:

```markdown

### Unique Value Proposition

|----------|-----------|-----------------|
```

### Step 6: 문서 동기화

가설 문서는 **로컬 파일**과 **외부 문서**를 동시에 생성하고, 이후 변경 시 항상 양쪽을 함께 업데이트한다.

#### 로컬 문서 (필수)
- **파일 경로**: `docs/hypotheses/[slug].md`
- 모든 가설의 Single Source of Truth. 항상 생성한다.
- `/validate-landing`, `/validate-judge` 등 후속 스킬이 이 파일을 참조한다.

#### 외부 문서 (선택적)
사용자가 Confluence, Notion 등 외부 도구 동기화를 요청하면:
- 같은 내용으로 외부 플랫폼에 페이지를 생성한다
- 외부 문서에는 체크리스트(가설 검증 계획)를 포함하여 진행 상황 추적이 가능하게 한다

#### 동기화 규칙
- **생성 시**: 로컬 md + 외부 문서를 동시에 생성한다.
- **변경 시**: 대화 중 내용이 변경(가설 수정, 검증 기준 변경, 판정 결과 추가 등)되면 **로컬 md와 외부 문서를 항상 함께 업데이트**한다. 한쪽만 업데이트하지 않는다.
- **상태 변경 시**: `/validate-judge` 결과로 상태가 바뀌면(🔬 검증 전 → ✅ GO / 🔄 PIVOT / ❌ KILL) 양쪽 모두 반영한다.
- 로컬 md 파일의 하단에 외부 문서 링크를 기록하여 추적 가능하게 한다:

```markdown
---

## 연결된 문서
- Confluence: [페이지 제목](URL)
```

### Step 7: 사용자 확인

생성된 문서를 보여주고 수정할 부분이 있는지 확인한다.
수정 요청이 있으면 반영 후 **로컬 md + 외부 문서 양쪽 모두** 저장.

## 소통 원칙

- **쉬운 말로 설명한다.** 전문 용어(인텐트 데이터, 앵커링 등)를 쓸 때는 반드시 한 문장으로 풀어서 설명한다. 사용자가 마케팅/비즈니스 전문가가 아닐 수 있다.
- **리서치는 적극적으로 한다.** "모르겠다"는 답은 대화의 끝이 아니라 리서치의 시작이다. 웹 검색, 경쟁사 분석, 벤치마크 조사를 수행하여 의사결정을 돕는다.
- **리서치 후에도 결정은 사용자가 한다.** 데이터를 제시하고 추천은 하되, 강요하지 않는다.
- **이미 아는 것은 다시 묻지 않는다.** 대화 흐름에서 이미 언급된 정보는 정리해서 확인만 받는다.

## 주의사항

- "이게 좋은 아이디어인지" 판단하지 않는다. 그건 데이터가 할 일이다.
- 답변이 모호하면 구체화 질문을 한다. 넘어가지 않는다.
- 가설 문장은 반드시 **측정 가능한 숫자**를 포함해야 한다.
- 리서치 결과를 제시할 때 출처/근거를 함께 보여준다.
