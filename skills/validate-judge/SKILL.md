# /validate-judge

수집된 트래픽 데이터를 입력하면, 벤치마크 기준으로 Go / Pivot / Kill 판정을 내린다.

## 트리거

사용자가 다음과 같이 말할 때:
- "데이터 분석해줘", "결과 판단해줘", "검증 결과 봐줘"
- "validate judge", "go or kill"
- `/validate-judge`

## 사전 조건

가설 문서(`docs/hypotheses/[name].md`)가 있으면 해당 문서의 검증 기준을 사용한다.
없어도 실행 가능 — 범용 벤치마크를 사용한다.

## 워크플로우

### Step 1: 데이터 수집

사용자에게 다음 숫자를 질문한다. 모르는 항목은 건너뛸 수 있다.

```
📊 데이터 입력

[필수]
1. 랜딩페이지 총 방문자 수:
2. CTA 전환 수 (이메일 등록 / 결제 시도 / 상담 신청 등):

[선택]
3. 총 광고비 (원):
4. 광고 클릭수 (CTR 계산용):
5. 광고 노출수:
6. 테스트 기간 (일):
```

### Step 2: 지표 자동 계산

입력된 데이터로 다음을 계산한다:

```
CVR (전환율)     = 전환수 ÷ 방문자수 × 100
CPA (전환당 비용) = 광고비 ÷ 전환수
CPC (클릭당 비용) = 광고비 ÷ 클릭수
CTR (클릭률)      = 클릭수 ÷ 노출수 × 100
```

### Step 3: 벤치마크 비교

#### 가설 문서가 있는 경우
→ 문서에 정의된 검증 기준(GO/PIVOT/KILL 전환율)을 사용한다.

#### 가설 문서가 없는 경우 → 범용 벤치마크 적용

**CTA 유형별 전환율 기준:**

| CTA 유형 | ❌ KILL | 🔄 PIVOT | ✅ GO |
|----------|--------|----------|-------|
| 이메일/웨이트리스트 | < 3% | 3~10% | > 10% |
| 가격표 클릭 | < 5% | 5~15% | > 15% |
| 사전결제 | < 1% | 1~5% | > 5% |
| 무료 체험 | < 5% | 5~15% | > 15% |
| 상담 신청 | < 2% | 2~8% | > 8% |

**광고 지표 참고 기준:**

| 지표 | 부정적 | 보통 | 긍정적 |
|------|--------|------|--------|
| Meta Ads CTR | < 1% | 1.5~2.5% | > 3% |
| Google Ads CTR | < 2% | 3.5~6% | > 6% |
| CPC (Meta) | > ₩3,000 | ₩1,000~3,000 | < ₩1,000 |

### Step 4: 판정

3단계 판정을 내린다:

#### ✅ GO
**조건**: CVR이 GO 기준 이상
**의미**: 사람들이 돈을 낼 의향이 확인됨. MVP 개발 착수.
**다음 단계**: 제품 개발 시작, 가설 문서 상태를 `✅ 검증 완료 (GO)`로 업데이트

#### 🔄 PIVOT
**조건**: CVR이 PIVOT 범위 안
**의미**: 관심은 있지만 전환이 부족. 메시지, 타겟, 가격 중 하나를 바꿔서 재실험.
**다음 단계**: 아래 진단을 수행한다.

PIVOT 진단 — 뭘 바꿔야 하는가:

```
광고 CTR 낮음 + CVR 낮음
→ 메시지(가치 제안) 자체가 안 먹힘. Problem/UVP 재정의 필요.

광고 CTR 높음 + CVR 낮음
→ 광고는 끌지만 랜딩이 설득 실패. 랜딩 카피/구조 수정.

광고 CTR 낮음 + CVR 높음
→ 랜딩은 좋은데 타겟이 잘못됨. 광고 타겟팅 변경.

CPA가 가격 대비 너무 높음
→ 단가 대비 고객 확보 비용 과다. 가격 인상 또는 채널 변경.
```

#### ❌ KILL
**조건**: CVR이 KILL 기준 미만
**의미**: 이 가설은 시장이 원하지 않음. 다음 아이디어로.
**다음 단계**: 가설 문서 상태를 `❌ 검증 완료 (KILL)`로 업데이트. 새 가설 시작.

### Step 5: 리포트 생성

**파일 경로**: `docs/hypotheses/[slug]-result.md`

**템플릿**:

```markdown
# [아이디어명] — 검증 결과

> 판정일: [YYYY-MM-DD]
> 판정: [✅ GO / 🔄 PIVOT / ❌ KILL]

## 입력 데이터

| 항목 | 값 |
|------|-----|
| 방문자 | [N]명 |
| 전환 | [N]건 |
| 광고비 | ₩[N] |
| 테스트 기간 | [N]일 |

## 산출 지표

| 지표 | 값 | 벤치마크 | 평가 |
|------|-----|----------|------|
| CVR | [N]% | GO: >[X]% | [✅/🔄/❌] |
| CPA | ₩[N] | - | - |
| CPC | ₩[N] | < ₩1,000 양호 | [평가] |
| CTR | [N]% | > 2.5% 양호 | [평가] |

## 판정

### [✅ GO / 🔄 PIVOT / ❌ KILL]

[판정 근거 2~3문장]

## PIVOT인 경우: 변경 제안

[진단 결과 기반 구체적 제안]

## 다음 단계

[판정에 따른 액션 아이템]
```

### Step 6: 가설 문서 업데이트

원본 가설 문서(`docs/hypotheses/[slug].md`)의 상태를 업데이트한다:
- `🔬 검증 전` → `✅ 검증 완료 (GO)` 또는 `🔄 PIVOT 필요` 또는 `❌ 검증 완료 (KILL)`

## 주의사항

- 300명 미만의 트래픽은 통계적으로 유의미하지 않다. 300명 미만이면 경고를 표시하되 판정은 수행한다.
- "설문 결과"나 "쓸 것 같다는 응답"은 받지 않는다. **실제 행동 데이터(클릭, 등록, 결제)만 유효하다.**
- PIVOT 판정 시 반드시 "뭘 바꿔야 하는지"까지 제안한다. 판정만 하고 끝내지 않는다.
- 감정이 아니라 숫자로 판단한다. 사용자가 "감이 좋은데요"라고 해도 숫자가 KILL이면 KILL이다.
